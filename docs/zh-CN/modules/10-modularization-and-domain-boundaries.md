# 模块化与领域边界拆分指导

## 文档目的

这份文档用于沉淀一套长期有效的架构指导思想：在不预设具体拆分结果的前提下，持续提升系统的模块内聚性、边界清晰度、可测试性和演进稳定性。

它不是实施计划，也不规定某个模块必须先拆、后拆，或必须演化成什么形态。它只回答一个问题：

**当系统继续增长时，我们应该如何定义、观察和维护模块与领域边界。**

## 适用范围

这份指导适用于整个控制面代码库，尤其适用于以下几类持续演进中的能力：

- 生命周期驱动的核心对象
- 配额、规则、限制与后台检查
- API、CLI、SDK 与内部服务之间的接线
- 后台任务、同步循环与状态修复逻辑
- 共享模型、错误语义与跨模块事件

## 核心判断

随着功能增长，系统复杂度的主要来源通常不是“代码量变多”，而是以下几种边界被不断混写：

- 领域语义边界
- 状态机边界
- 决策边界
- 副作用边界
- 接口边界

当这些边界混在同一个函数、同一个服务对象或同一个任务循环里时，常见后果是：

- 一个改动需要同时理解太多上下文
- 测试只能走超长链路，难以稳定定位问题
- 同一条规则在多个入口重复实现
- 状态变化和资源操作交织，难以推断一致性
- 后续即使想拆分服务，也很难找到自然切口

因此，模块化的目标不是机械地“多拆几个目录”，而是让每一层复杂度都在自己的边界内闭合。

## 基本原则

## 1. 先拆领域，再谈拆服务

领域边界优先于部署边界。

如果一个系统在概念上还没有形成稳定的职责分工，那么过早拆成多个服务，只会把原本的进程内混乱搬到网络边界两侧。这样不会降低复杂度，只会把问题变得更难观测、更难调试。

因此，优先目标始终是：

- 先明确模块职责
- 先抽清调用接口
- 先稳定状态语义
- 再视情况决定是否需要物理拆分

## 2. 模块以“领域能力”定义，不以技术层定义

一个自然模块应该围绕某类稳定的业务问题组织，而不是围绕某种技术手段组织。

更合适的模块定义通常是：

- 某个核心对象的生命周期
- 某类规则与授权决策
- 某类资源消耗与结算语义
- 某类对外接入面与协议转换
- 某类后台协调与一致性修复

不合适的模块定义通常是：

- “把所有 cron 都放一起”
- “把所有数据库操作都放一起”
- “把所有 Kubernetes 相关代码都放一起”
- “把所有 API 路由都当成一个模块”

技术实现方式可以被多个领域共享；领域语义本身不应被技术分层掩盖。

## 3. 决策与执行分离

系统中的很多复杂度，并不来自执行动作本身，而是来自“是否应该执行”。

一个健康的边界通常会把这两类问题分开：

- 某个模块负责给出判断、授权、限制或建议
- 另一个模块负责根据该判断去执行状态变更或外部副作用

这样做的价值在于：

- 规则更容易单测
- 执行动作更容易复用
- 后续接异步任务、事件或远程调用时，迁移成本更低

## 4. 状态变更与副作用分离

当代码同时承担下面两种职责时，系统通常会变脆：

- 更新领域状态
- 发起外部系统调用

例如：

- 改数据库状态
- 调 Kubernetes
- 写 usage ledger
- 触发后台停止动作

这些行为可以在同一条业务链路上发生，但它们不应该在语义上被视为同一种事情。领域状态描述“系统认为自己处于什么状态”；外部副作用描述“系统尝试让外部世界也到达对应状态”。

边界清楚之后，失败补偿、重试、幂等和 reconcile 的设计才会变得自然。

## 5. 模块通过显式接口协作，不通过隐式共享知识协作

一个模块不应该依赖另一个模块的内部表结构、内部状态枚举或内部推导过程，除非这些内容已经被正式纳入边界契约。

更稳定的协作方式是：

- 显式的 service interface
- 明确的 command / query 输入输出
- 收敛后的领域错误
- 语义稳定的事件或结果对象

不稳定的协作方式是：

- 直接跨模块拼 SQL 或读取内部字段
- 在多个入口各自复制一遍规则
- 用布尔值和魔法字符串隐式传递复杂判断

## 6. 共享代码应当最小化且中性

共享层只适合放以下内容：

- 基础类型
- 稳定错误语义
- 通用时间、分页、标识符等工具
- 真正跨领域且无业务倾向的公共函数

共享层不应该成为“暂时不知道放哪，就先放 shared”的缓冲区。否则共享层会反向变成新的耦合中心。

## 模块单元的理想形态

一个成熟模块，通常应具备以下特征：

- 有清楚的核心对象或核心概念
- 有相对闭合的状态语义
- 有明确的输入、输出和错误边界
- 能在不理解全系统的情况下被单独测试
- 能解释“这个模块负责什么”以及“它明确不负责什么”

如果一个模块必须依赖大量外部上下文才能理解，则说明边界仍然过薄或职责仍然混杂。

## 边界定义的五个层次

为了避免只在目录层面做表面拆分，可以从五个层次同时判断边界是否成立。

### 1. 词汇边界

模块内部是否使用一套稳定、收敛的术语。

如果同一概念在不同模块里被反复换名，说明领域语言还不稳定。

### 2. 规则边界

某条规则是否有唯一归属。

如果一个限制条件分别散落在 API、service、task 和 reconcile 多处实现，那么规则还没有真正归入某个模块。

### 3. 状态边界

状态机是否属于某个稳定的对象或流程。

如果一个状态需要多个模块共同“解释”才知道是什么意思，那么状态边界就还不清楚。

### 4. 数据边界

模块是否可以围绕自己的数据视图完成主要工作，而不是频繁窥探其他模块的内部存储结构。

### 5. 副作用边界

对外系统调用、资源创建、停止、修复、回收等动作，是否能被清楚地归属给某个执行者。

## 面向未来的阶段演进

这套指导思想可以自然经历几个阶段。阶段的作用是帮助判断“当前更适合做什么”，而不是规定固定时间表。

### 阶段一：模块化单体

这个阶段的重点是：

- 代码仍可运行在同一进程中
- 模块之间优先走进程内接口
- 数据存储可以仍然集中
- 重点是把职责、规则和状态语义先收敛清楚

这是系统建立边界意识的起点，也是最适合强化测试和清理耦合的阶段。

### 阶段二：单元化协作

当模块边界逐渐稳定后，可以开始把“单体中的模块”看成“可以被独立观察和替换的服务单元”。

这个阶段的重点不是物理拆分，而是：

- 接口进一步稳定
- 模块输入输出更像契约
- 后台流程更像协调而不是硬编码串联
- 事件、命令、查询的语义逐渐收敛

此时即使仍在同一进程里，模块之间也已经具备了更强的自治性。

### 阶段三：物理拆分的准备态

当某些模块的节奏、可靠性要求、资源需求或演进方向明显不同，才进入“是否需要独立部署单元”的讨论。

这个阶段关注的是：

- 哪些边界已经足够稳定
- 哪些调用具备远程化条件
- 哪些一致性要求适合通过异步协调解决
- 哪些观测指标已经可以支持跨单元运行

进入这一阶段，并不意味着必须立即拆分；它只是说明系统已经具备了“可拆”的结构条件。

## 如何判断某个边界是否健康

以下现象通常说明边界是健康的：

- 一个模块的改动主要局限在该模块内部
- 单测可以围绕某个领域规则直接书写
- 集成测试主要验证模块之间的接线，而不是补规则缺口
- 新增入口时，规则可以通过现有接口复用
- 后台任务更多是协调者，而不是第二套业务实现

以下现象通常说明边界仍然模糊：

- 每加一个入口都要重新复制一遍判断逻辑
- API、任务循环和修复流程分别维护自己的规则版本
- 一个模块的测试必须先搭好多个不相关模块才能启动
- 代码改动经常横跨大量目录且无法解释其必要性
- “先放 service 里”变成所有复杂度的最终归宿

## 对测试路径的要求

更清晰的领域边界，应该直接改善测试结构。理想状态下，测试路径也应当与边界层次对应：

- 规则测试：验证领域判断是否正确
- 状态测试：验证状态迁移是否闭合
- 协调测试：验证模块之间的编排和接线
- 集成测试：验证真实副作用与外部系统交互
- 修复测试：验证 reconcile、补偿和幂等行为

如果大量测试只能通过“从最外层一路跑到最深处”来完成，通常意味着领域接口还不够清楚。

## 后台任务的角色定位

后台任务不应成为新的业务中心。

更合适的定位是：

- 推进已经定义好的领域流程
- 检查状态与现实是否偏离
- 驱动补偿、修复与回收
- 执行周期性检查与协调

它们不应该承担的职责是：

- 独立定义一套与主链路不同的业务规则
- 成为唯一知道系统真实语义的地方
- 用“定时扫一遍”替代清晰的边界设计

## 面向实际演进的取舍

在系统继续发展时，应优先追求下面几种改进：

- 让一个领域问题只在一个地方被真正定义
- 让一个模块可以被局部理解、局部测试、局部替换
- 让接口表达语义，而不是泄露内部实现
- 让跨模块协作依赖契约，而不是依赖记忆

不必追求下面这些表面上的“整洁”：

- 为了目录整齐而提前拆成很多服务
- 为了所谓解耦而把简单同步调用强行改成异步
- 为了抽象统一而过早引入过多中间层

架构清晰度来自边界语义，而不是来自文件数量、服务数量或图画得有多漂亮。

## 最终目标

这份指导希望长期推动系统形成以下特征：

- 模块内部高内聚，模块之间低耦合
- 每个领域边界都有稳定的语言、规则和状态语义
- 业务决策、状态变更和外部副作用各有明确归属
- 测试结构能够自然映射系统结构
- 当未来需要单元化、远程化或独立部署时，系统可以顺势演进，而不是被迫重写

换句话说，真正重要的不是“是否拆服务”，而是：

**系统是否已经先成为一个边界清楚的系统。**
